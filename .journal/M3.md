## Goals

Relocate the Storage interface out of `pkg/raft` (where Phase 2 had it as
an `RFC-0003` stub) into its own `pkg/storage` package, freeze the 10-method
surface (Append/TruncateSuffix/Entries/Term/FirstIndex/LastIndex +
SaveHardState/LoadHardState + Snapshot/Restore), ship an in-RAM impl at
`pkg/storage/memory`, and prove conformance with a public test harness at
`pkg/storage/storagetest`. Discharges ROADMAP Phase 3 Success Criteria 1–4
and REQ-STOR-01/02/07.

## ADRs introduced

- **ADR-0005** (`docs/adr/0005-storage-interface-freeze.md`) — Storage
  interface freeze. Three deferred decisions from RFC-0003 resolved:
  (a) interface home is `pkg/storage` (not `pkg/raft`); (b) Snapshot /
  Restore live on `StateStorage` (option-a, mirrors `StateMachine` for v2
  symmetry); (c) `ErrSnapshotUnsupported` sentinel home is `pkg/raft`
  (inverted from the LLD §4 sidebar's original direction — see surprise
  #1). Additionally records a Consequences bullet that `commitIndex` lives
  in `Node.commitIndex` only and `HardState.Commit` is a best-effort
  hint, absorbing ROADMAP Phase 3 SC4's wording mismatch without
  superseding ADR-0004.

## Decisions made (non-ADR)

- **storagetest single public entry point (option-a).** `RunConformance(t,
  Factory)` is the only exported function. Mirrors stdlib house style
  (`testing/quick.Check`, `iotest`, `fstest.TestFS`). Leaves room for an
  additive `RunConformanceWithOptions` later without breaking the v1 API.
  Recorded in 03-03 plan + summary; not ADR-worthy on its own.
- **External test package (`memory_test`, not `memory`)** for the
  conformance wiring — proves the public surface of `pkg/storage/memory`
  is sufficient; no internal hooks needed.
- **Per-entry `Entry.Data` deep-copy on BOTH `Append` and `Entries`** in
  `pkg/storage/memory`. Bare `append(m.entries, entries...)` would alias
  the caller's backing array; both directions use a `for _, e := range ...`
  loop with `make([]byte, len(e.Data))` + `copy`. Conformance sub-tests
  13 (EntriesCallerCanMutate) and 14 (AppendCallerCanMutate) both passed
  on first run, validating the decision.

## What surprised us

1. **Sentinel home inverted on first build.** Plan 03-01 (and the LLD §4
   sidebar) originally said `ErrSnapshotUnsupported` lives in
   `pkg/storage`, re-exported by `pkg/raft`. But `pkg/storage` already
   imports `pkg/raft` for `Entry` / `HardState` / `Index` / `Term`, so
   `pkg/raft` importing `pkg/storage` produces a true import cycle.
   `go build ./...` failed with `import cycle not allowed`. The fix was
   to invert the direction: sentinel home is `pkg/raft/errors.go`,
   `pkg/storage` re-exports as `var ErrSnapshotUnsupported =
   raft.ErrSnapshotUnsupported`. `errors.Is` resolves identically against
   either name. ADR-0005 §3 documents the inversion; LLD §4 sidebar
   amended in the same atomic PR (Working Agreement 4). Lesson: sentinel
   homes follow the deeper-graph package — if A imports B for types,
   sentinels that A also needs must live in B (or in A directly), not be
   exported from A's importer.
2. **Caller-mutation probes passed on first run.** Sub-tests 13 and 14
   exercise both directions of the deep-copy contract. Both green on the
   first `go test -race` run — confirms plan 03-02 correctly anticipated
   the contract and deep-copied `Entry.Data` on both `Append` (write) and
   `Entries` (read) paths. The temptation to copy only on Append (caller
   "owns" the input) would have failed sub-test 13 immediately; doing it
   symmetrically was the right call.
3. **ROADMAP SC4 wording mismatch absorbed cleanly into ADR-0005.** The
   roadmap text said SC4 = "commitIndex policy recorded as ADR-0004
   amendment", but ADR-0004's scope is strictly the single-mutex policy;
   amending it with a storage-layer rule would have muddied its purpose.
   Recording the commitIndex-in-memory rule as a Consequences bullet on
   ADR-0005 instead satisfied the underlying requirement (rule is
   recorded in a phase-3 ADR) without distorting ADR-0004. The wording
   mismatch was caught during 03-RESEARCH; the resolution is documented
   in ADR-0005 §Consequences.
4. **storagetest's 14-sub-test count matched the LLD §3 invariant count
   exactly.** No invariant got dropped, none got duplicated. The
   one-`t.Run`-per-invariant pattern gives precise failure localization
   in Phase 8 when the file-impl runs the same harness.

## Seeds for Phase 4 (FakeClock + inproc Hub)

1. The Storage interface is now stable — Phase 5 will be the first phase
   to wire `Node` to `Storage`. Phase 4 should NOT touch
   `pkg/storage/...`; it's clock + transport only.
2. `pkg/storage/memory` is the test substrate that Phase 5+ election /
   replication tests will use. The Storage carries no internal locking
   (single-mutex policy, ADR-0004) — callers hold `Node.mu`. Phase 4's
   transport hub follows the same pattern: no internal lock; the Node
   serialises access.
3. `storagetest.RunConformance` is the public-API template for future
   per-component conformance harnesses. If Phase 4's inproc Hub grows a
   public contract, mirror the `RunConformance(t, Factory)` shape and
   sit it beside the package (not under `internal/`).
4. ADR-0005 records that `commitIndex` lives in `Node.commitIndex` only;
   `HardState.Commit` is a best-effort hint. Phase 4 doesn't need this
   today; Phase 5 (election) MUST read this before touching `HardState`.
5. The drift gate now covers `pkg/storage` and `pkg/storage/storagetest`.
   Any phase that adds new public types to those packages — or to
   `pkg/raft` — must run `make lld-drift-update` and commit the golden
   in the same PR.
