> Land the election sub-protocol (follower + candidate roles) behind a
> single mutex with deterministic timeouts, durable vote persistence, and
> a chaos-replayable at-most-one-leader-per-term invariant. Phase 6

> spine that the entire rest of the system relies on.

- `pkg/raft/maybe_step_down.go` — `maybeStepDownLocked` + epoch token
  (`stepDownEpoch`). The epoch token is the TOCTOU-free step-down

- `pkg/raft/config.go` — `Config` (`ID`, `Peers`, durations, `Seed`,

  did not.
- `pkg/raft/follower.go` — `tickFollowerLocked` +

## ADRs introduced

- **ADR-0008** (`docs/adr/0008-step-event-loop-and-ready.md`) — step()
  event-loop + Ready() decoupling for the Raft core state machine.
  Locks: `Step` is the single inbound entry (synchronous, holds the
  mutex for the term-first funnel only); per-role handlers MUST NOT
  inline term checks; outbound messages and `HardState` updates queue
  on the node and drain via `Ready()`; the `stepDownEpoch` token (folded
  in from the optional ADR-0010 slot per RESEARCH guidance) makes
  step-down TOCTOU-free by tagging every pending outbound with the
  epoch it was queued under. References ADR-0004 (single mutex).
- **ADR-0009** (`docs/adr/0009-per-node-rng-mixing.md`) — Per-node RNG
  mixing for randomised election timeouts. Locks: `FNV-1a(nodeID) ⊕
  splitmix64(Seed)` seeds a `math/rand/v2.PCG` per node; Seed=0
  degrades to `clk.Now().UnixNano() XOR FNV(nodeID)` (entropy routes
  through `Config.Clock` so no `time.Now()` leaks into `pkg/raft`);
  same Seed + different node IDs produces ≥95% differing draws over
  100 timeouts (SC2 lock). References P1-4 (RNG global source).

## What surprised us

1. **The local `pkg/raft.Storage` subset interface was unavoidable.**
   ADR-0005 froze `pkg/storage.Storage`, and Phase 3 made `pkg/storage`
   import `pkg/raft` for the `Entry` / `HardState` / `Index` / `Term`
   types — so `pkg/raft` cannot import `pkg/storage` back. Plan 05-01
   declared a minimal local subset (`LoadHardState` /
   `SaveHardState` / log accessors actually consumed) that
   `pkg/storage.Storage` structurally satisfies. The duplication
   surfaced cleanly in the LLD-drift golden; we backpatched §3 with an
   explicit divergence note and flagged Phase 7 ADR reconciliation
   (move shared types to a leaf, or canonicalise the subset). The
   surprise wasn't the cycle — RESEARCH §Pattern 3 anticipated it — but
   that the cleanest path through it was a documented duplication
   rather than a refactor.
2. **`Config.Clock` was a mid-execution add (commit `5caf96a`).** Plan
   05-02 originally implemented the RNG zero-seed fallback with
   `time.Now()` directly; `scripts/check-no-time-now.sh` (Phase 4
   gate) caught it immediately. The fix added `Clock` to `Config` and
   routed the fallback through `clk.Now()`. This validates the Phase 4
   gate as load-bearing: without it, the leak would have shipped
   silently, and the seed-determinism guarantee would have been a lie
   in production. The Phase 4 lint paid for itself on its first real
   test.
3. **Equal-term observations MUST NOT bump `stepDownEpoch`.** The
   epoch filter in `Ready()` uses strict equality
   (`pm.epoch == n.stepDownEpoch`) rather than `>=`. Initial sketches
   used `>=`, which incorrectly invalidated same-term heartbeats and
   broke vote-response delivery. The strict-equality rule is now
   pinned by `TestReadyEpochFilterDropsStaleMessages` and codified in
   ADR-0008. The asymmetry — "only true step-downs invalidate" — is
   the correct semantics but is easy to get wrong.
4. **Three-layer SC5 enforcement felt like overkill but caught a real
   bug.** The first plan-05-04 draft of the driver had `Send` run
   before `SaveHardState`; the raftdebug invariant fired on the first
   `-tags raftdebug` run. Without the invariant, the OrderingStorage
   would have caught it; without OrderingStorage, the chaos suite
   would have caught it (eventually) via a violated
   at-most-one-leader-per-term invariant on a sufficiently nasty seed.
   But the three-layer setup caught it at the cheapest possible point
   (single invariant check vs 1000 seeds × 200 ticks). Worth the
   construction cost.
5. **The 1000-seed chaos suite was fast.** Under 5 seconds in CI on
   the M1 reference, exactly as RESEARCH projected. The
   `Cluster.Tick` 2ms-quiesce-between-passes optimisation halved
   convergence latency vs the naive "one pass per call" implementation
   — same-tick replies now land on receivers before the next drain.
6. **Exported `raft.TestNode` in a non-`_test.go` file is a code
   smell, but the alternative was worse.** `internal/raftest` imports
   `raft.NewTestNode` at runtime (not from a `_test.go` file), so the
   handle must be exported and reachable. Phase 7 deletes it when
   `raft.New` / `raft.Node` land. The LLD-drift golden now records
   the leak explicitly; Phase 7 will catch any further drift.

## Seeds for Phase 6 (Replication + Commit (Leader))

1. **`tickLeaderLocked` is the first plan.** Heartbeat fan-out at the
   `HeartbeatInterval` cadence; the no-op stub in `leader_stub.go`
   already exists, so the first commit is the smallest possible diff.
   `Config.Validate`'s `HeartbeatInterval*3 <= ElectionTimeoutMin`
   invariant guarantees a single dropped heartbeat does not provoke
   re-election.
2. **`nextIndex` / `matchIndex` maps are wired.** `becomeLeaderLocked`
   already populates them per-peer; replication's `AppendEntries`
   fan-out reads from `nextIndex`, writes back on response.
3. **`handleAppendEntriesRespLocked` is a stub awaiting bodies.** The
   match-or-rewind decision (REPL-04..REPL-06) and the
   majority-commit advance (REPL-07) both land here.
4. **The Ready() seam likely suffices for Phase 6.** `pendingMsgs +
   pendingHS` cover heartbeats, append fan-out, and HardState updates.
   If commit/apply events need surfacing, expand the return tuple
   rather than introducing a new channel — keep the synchronous
   single-drain shape.
5. **Chaos suite extends naturally.** `TestAtMostOneLeaderPerTerm_Chaos`
   becomes the harness for `TestNoLogDivergence_Chaos` in Phase 6 — same
   1000-seed × 200-tick × 5-node setup, new invariant. Reuse the
   `Cluster.tickOnce` driver verbatim.
6. **Watch for P0-4 full retirement.** The append-then-respond ordering
   for entries (separate from the vote ordering already retired) is
   Phase 6's P0-4 obligation. The three-layer SC5 enforcement pattern
   (driver discipline + raftdebug invariant + OrderingStorage) is the
   template; promote `OrderingStorage` to also check
   `Append-precedes-AppendEntriesResponse`.
