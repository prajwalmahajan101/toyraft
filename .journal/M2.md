## Goals

Freeze the public type surface (`pkg/raft/types.go`), land the in-memory `Log`
with the six FOUND-02 methods, ratify the single-mutex state machine policy in
an ADR, and ship the SC6 byte-for-byte drift check so every later phase is
guarded against quietly diverging from `docs/LLD.md`. Original RESEARCH.md
scoped this at 4 plans; the working plan-set expanded to 7 via two
RFC-driven bring-forwards (RFC-0002 fuzz, RFC-0003 Storage signatures).

## ADRs landed

- **ADR-0004** (`docs/adr/0004-single-mutex-state-machine.md`) — single-mutex
  state machine policy. Discharges the forward reference in
  `docs/CONCURRENCY.md §2` (back-patched). ADR numbering uses 0004 (not 0001)
  because Phase 1.1 inserted 0002 (`bring-ci-forward`) and 0003 — this is the

## RFCs landed

- **RFC-0002** (`docs/rfc/0002-fuzz-coverage-forward.md`) — **Accepted.**
  Justifies pulling fuzz coverage for the in-memory `Log` surface forward
  from Phase 11/12. Bounded strictly to the six FOUND-02 methods; CI-gated
  to pull_request events to keep main-branch CI fast. Implementation in
  plan 02-06.
- **RFC-0003** (`docs/rfc/0003-storage-interface-stub-in-phase-2.md`) —
  **Accepted (approve-stub path).** Storage interface signatures from LLD §3
  brought forward to Phase 2 as stubs returning `ErrStorageNotImplemented`.
  Phase 3 inherits the unchanged interface shape and provides the real impl
  in `pkg/storage/memory`. Interfaces declared: `Storage`, `LogStorage`,
  `StateStorage`. `ErrSnapshotUnsupported` is NOT declared in this phase —
  LLD §3 lookup found no such sentinel; Phase 3 will own it when snapshot
  semantics are designed.

## Bring-forward decisions

Both bring-forwards expanded the original 4-plan scope to 7 plans / 4 waves.
Documented here as intentional revisions (informed by RESEARCH.md Q5 and the
02-RESEARCH addendum), not deviations.

- **Fuzz coverage from Phase 11/12 → Phase 2** (plan 02-06, RFC-0002):
  bounded to `*Log` surface only. Out-of-scope: Node / Storage / Transport /
  multi-node — those remain Phase 11/12. Rationale: Log invariants are
  load-bearing; catching bugs at the unit-fuzz layer is dramatically cheaper
  than finding them in Phase 11's chaos suite. **This bring-forward
  immediately paid off:** the macOS CI fuzz run on the Phase-2 closing PR
  surfaced a div-by-zero in the fuzz harness arithmetic
  (`byte(expectedLast)` truncates to 0 when `expectedLast >= 256`). Fixed
  in commit `64730cb`; production code unaffected.
- **Storage interface signatures from Phase 3 → Phase 2** (plan 02-07,
  RFC-0003): stub-only (signatures + sentinel, no impl). Phase 3 retains
  exclusive ownership of `pkg/storage/...` sub-packages, the
  `storage/memory` impl, and the `storage/storagetest` conformance harness.
  Rationale: Phase 5 (election) needs `SaveHardState` / `LoadHardState`
  symbolically; Phase 3's scope narrows to impl quality alone.

## What surprised us

- The fuzz bring-forward (RFC-0002) paid off **on the closing PR itself.**
  macOS fuzz runners found `byte(0)` div-by-zero in the harness within 14
  seconds. If we'd deferred fuzz to Phase 11/12 as originally planned, that
  bug would have lurked in `testdata/fuzz/` corpus seeds for months. Strong
  validation that the bring-forward was the right call.
- The 7-plan expansion (vs. the original 4-plan RESEARCH.md scope) was
  driven entirely by two RFCs, both surfaced during phase planning before
  any code landed. The RFC-as-scope-adjustment mechanism worked exactly as
  intended — no "we should have planned for X" hindsight.
- Branch-protection-via-API (Phase 1.1) made the closing-PR phase smooth:
  no need to edit a settings file mid-phase, the contexts are immutable
  unless someone explicitly `gh api PUT`s. The 7-context query in
  Task 3 / Task 4 took 1 second and the baseline was provably untouched.

## Seeds for Phase 3 (Storage Interface + Memory Impl)

1. `pkg/raft/log.go`'s `Term(idx)` deliberately matches the LLD
   `LogStorage.Term` contract — Phase 3's `memory.Storage.Term` can delegate
   via `return mem.log.Term(idx)`.
2. The single-mutex policy (ADR-0004) means Phase 3's `memory.Storage`
   doesn't need internal locking either — caller (Node) holds `Node.mu`.
   Document this in Phase-3's plan-context.
3. `Storage.Append` signature in LLD uses `[]Entry`; `Log.Append` is
   variadic. Phase 3's `memory.Storage.Append` should spread:
   `mem.log.Append(entries...)`.
4. Phase 2 took the **approve-stub path** for RFC-0003, so Phase 3 inherits
   the Storage interface shape unchanged — only stub method bodies need
   replacing. The `ErrStorageNotImplemented` sentinel can stay (used for
   "not yet wired" branches) or be removed once all methods have real impls.
5. `ErrSnapshotUnsupported` is **not** in Phase-2's types — Phase 3 owns
   that sentinel when snapshot semantics are designed.
6. The SC6 drift-check golden is generated AFTER types stabilise. When
   Phase 3 lands new public types in `pkg/storage`, that phase will need
   its own golden (or extend ours to include `./pkg/storage`).
